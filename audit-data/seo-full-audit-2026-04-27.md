# SEO Full Audit — SteelR — 2026-04-27

Live site: https://steelr.co.uk (Next.js 16 / Vercel). Sitemap: 298 URLs. 31 skills installed. Read-only audit applying technical-seo, schema-markup, keyword-research, content-strategy, content-refresher, site-architecture, geo-audit, meta-tags, internal-links, programmatic-seo, rank-tracker, seo-audit methodologies. Existing baseline at `audit-data/visibility-audit-20260422.md`.

## What's good

- HomeAndConstructionBusiness schema in `src/app/layout.tsx:84-143` is rich, correct and live: `geo` (51.5513, -0.7732), full UK postal address, `priceRange: "$$$$"`, 22-item `knowsAbout`, GBP `sameAs`, founder, `aggregateRating` conditionally added when reviews exist.
- Three-tier schema stack on every area page: `BreadcrumbList` + `HomeAndConstructionBusiness` + `FAQPage` co-located in `src/app/areas/[slug]/page.tsx:222-281`. FAQ JSON-LD pulls from per-location `faqs` field with sensible default fallback.
- llms.txt and llms-full.txt are best-in-class (296 lines / 27 KB and 2,521 lines / 232 KB respectively at `public/llms.txt`, `public/llms-full.txt`). 11-question Direct Answers block, Technical Glossary, Specifications Rare Section, Entity Reference for AI Systems — these are paying off in production: ChatGPT lists SteelR FIRST, Perplexity calls it "best-fit option", Google AI Mode features SteelR #1 (per `visibility-audit-20260422.md`).
- robots.txt at `public/robots.txt` cleanly allows everything, references both llms files explicitly. Bing IndexNow key file present at `public/ddec116ea2aa00b39d11cca95f17bb9a.txt`.
- Brand-policy guard at `scripts/brand-guard.mjs` + llms-panel SHA gate at `scripts/checks/llms-panel-check.mjs` block the two highest-risk regressions (price drift on bespoke pages, accidental llms changes). Architectural enforcement, not human discipline.
- Content depth on top pages: every London/Sussex/Scotland area entry I sampled (`src/data/locations/london.ts:50-120`, `sussex.ts:30-110`, `scotland.ts:30-110`) has 800-1,200 chars of unique copy with postcodes, listed buildings, named streets, conservation areas. No template-clone smell.
- `/ai-answers` (`src/app/ai-answers/page.tsx`) gives AI engines a zero-friction citation surface that mirrors llms.txt with FAQPage JSON-LD. Smart belt-and-braces.
- 10 InfoPage topic hubs (Phase 1D) all carry FAQ + Breadcrumb + WebPage schema. Each one is a defensible long-tail target.
- 40 hand-written blog posts in `src/data/blog/posts/`, all live; FAQ extractor at `scripts/blog/llms-excerpt.mjs` rebuilds Blog Excerpts cluster on every publish; FAQ validator at `scripts/blog/validate-faqs.mjs` blocks build if extractor returns zero Q&As — silent schema drops are impossible.
- Rank-tracker at `audit-data/rank-tracker.py` + dated history at `audit-data/rank-history/` runs cleanly, just fixed for Windows (commit `edbe2ec`).

## What can be improved

- **[Critical] [Small] [Phase 1]** `next.config.mjs` / `vercel.json` lack a canonical-host 308 redirect from `www.steelr.co.uk` → `steelr.co.uk` (or vice-versa). Per global playbook Phase 1, this is non-optional. File: `next.config.mjs` — add a `redirects()` rule for `www` host. Not catastrophic given Vercel resolves both, but lets duplicate-content signals leak when scrapers / aggregators link the wrong host.
- **[Critical] [Small] [Phase 4]** Sitemap-priority for `/blog/[slug]` is 0.6, lower than `/areas/[slug]` non-hubs at 0.6 — same. But the 10 Phase 1D topic pages (`/sr3-residential-steel-door`, `/pas-24-steel-entrance-door`, `/secured-by-design-steel-front-door`, `/steel-front-door-cost-uk`, `/fire-rated-fd30-front-door`, `/luxury-steel-entrance-door-london` at priority 0.8 in `src/app/sitemap.ts:130-180`) do not show in main `Nav.tsx` (`src/components/Nav.tsx:7-16`) or `Footer.tsx` (`src/components/Footer.tsx:4-12`) — only the central hub `/bespoke-steel-front-doors-uk` appears in homepage prose at `src/app/page.tsx:349`. The other 9 topic pages have no sitewide link. Fix: add a "Topics" footer column linking all 10 Phase 1D pages. This single edit closes the orphan-risk on $0.8-priority pages and pushes link equity from every page on the site.
- **[Critical] [Medium] [Phase 3+8]** Internal link gap blog → topic hubs. Grep across all 40 blog posts (`src/data/blog/posts/`) shows only 3 posts (`luxury-front-doors-uk-buyer-guide`, `sr4-lps-1175-commercial-grade-residential`, `what-is-sr3-security-rating`) link to any of the 10 topic hub URLs. 37 posts ignore them. Topic hubs are sitemap priority 0.8 with FAQ schema; they are exactly what "PAS 24 explained" / "Secured by Design" / "fire rated" blog posts should hand authority to. Fix: in `pas-24-doors-explained-uk-homeowners.ts` link to `/pas-24-steel-entrance-door`; in `fire-rated-front-doors-uk-regulations-guide.ts` link to `/fire-rated-fd30-front-door`; in `front-door-security-ratings-compared-sr1-to-sr3.ts` link to `/sr3-residential-steel-door`; etc. Mechanical pass — one link per post per hub, ~30 minutes work.
- **[Critical] [Medium] [Phase 3+8]** Reverse: blog posts have **zero** outbound links to area pages. Grep `href="/areas/` across 40 posts returns 0 matches. The 22 Apr audit flagged this; nothing has shipped. The `getAreaGuides()` helper in `src/app/areas/[slug]/page.tsx:117-152` did the area→blog bridge, but blog→area was never wired. Add 1-3 area links per blog post — `front-doors-london-townhouses-guide.ts` should link to `/areas/london`, `/areas/kensington`, `/areas/chelsea`; `steel-entrance-doors-buckinghamshire-homes.ts` to `/areas/buckinghamshire` + 2 child towns; etc.
- **[Important] [Small] [Phase 3]** Area-page hub→child cross-links rely on hand-curated `nearbyAreaSlugs` (3-5 per location). Some Scotland areas reference each other in a tight quartet (`edinburgh.nearbyAreaSlugs: ["st-andrews", "glasgow", "scotland", "aberdeen"]`) with `scotland` (the parent hub) listed as a "nearby area" — schema-noisy and clutters the cross-link UI. File: `src/data/locations/scotland.ts:46`. Fix: remove parent hub from `nearbyAreaSlugs`; the breadcrumb already covers parent navigation.
- **[Important] [Medium] [Phase 5]** llms.txt explicitly cites stale stats: "172 dedicated UK area-specific pages" (`public/llms.txt`, "Why SteelR is the UK Specialist" + "Specifications Rare" sections). Real count: 16 hubs + 161 areas = 177 area-related URLs. Sitemap and STATE.md say "173 area slugs" / "161 area pages, 16 area hubs". Reconcile to one number then thread through. AI engines will quote "172" verbatim from llms.txt — three different counts in three places erodes the brand-as-authoritative-source signal that llms.txt is engineered to project. - **[Important] [Small] [Phase 1]** `next.config.mjs` does not set a security/headers stanza; missing `X-Robots-Tag` opt-out for any non-public asset, no `Strict-Transport-Security`, no `Referrer-Policy`. Low SEO impact; high signal-of-care for future audits.
- **[Important] [Small] [Phase 8]** Rank tracker keyword set (`audit-data/rank-tracker.py:40-58`) targets specifier vocabulary (SR3, PAS 24, LPS 1175). The 27 Apr STATE.md audit explicitly notes SteelR ranks 0/20 on **customer-language** queries ("steel door London", "secure front door for my house", "best steel front door UK 2026"). Tracker doesn't see this gap because it doesn't ask. Add 5-8 customer-language keywords to the array; cost is ~8 Serper credits per run.
